What is D3?

D3 stands for Data-Driven Documents. It's a JavaScript Library that focuses on reducing the redundancies of JS and adding functionality to the language that is data-driven. It's a data visualization library that focuses on handling and manipulating data from within the Javascript file, an external csv or tsv or even a live data source. D3 focuses on creating powerful and engaging data visualizations within a minimum amount of headache.

D3 brings data to life with by incorporating language and syntax that allows JS to play nicely with HTML, CSS, SVG. It’s a library that allows data to do the speaking and issuing of commands on how web elements are displayed. Instead of telling D3 how to do something, tell D3 what you want

One of the main features of D3 is how it trims fat in JS. We'll look two sets of code that do the same thing. The first set is written in plain JS and the other is written with D3.

var chart = d3.select(".chart");
var bar = chart.selectAll("div");
var barUpdate = bar.data(data);
var barEnter = barUpdate.enter().append("div");
barEnter.style("width", function(d) { return d * 10 + "px"; });
barEnter.text(function(d) { return d; });

This is what that same code looks like in D3

d3.select(".chart")
.selectAll("div")
.data(data)
.enter().append("div")
.style("width", function(d) { return d * 10 + "px"; })
.text(function(d) { return d; });

The power of D3 comes from its ability to group similar elements and vastly reduce the lines of code and number of variables that have to be written. What’s beautiful about this code isn’t how it’s cut down, but how it’s grouped. D3 focuses on grouping similar page elements together so they’re easier to work with. What does this mean? It means that if you mess something up, you don’t have rewrite your entire code.

D3 as a visualization tool creates simple charts and graphs by itself. To create more dynamic, visualizations, D3 enlists the help of SVG, a markup language that allows for the creation of two-dimensional graphics in a vector image format that supports interactivity and animation. SVG uses similar syntax to CSS and HTML. For example, instead of "background-color" in CSS, SVG uses "fill."

D3 generalizes. And by generalizing, we are able to group things together so they’re easier to work with. What is the end product of this? We have a few lines of JavaScript that reach across HTML, CSS, SVG and JS to create web elements. One way to look at it is that you are writing with multiple languages at once.

We'll cover how D3 conducts these generalizations and works with these groups in the next section.

How D3 works

We'll delve deeper into the key features of D3 in this section. But first, let's go over how to load the library

Loading D3

All you have to do is point your script tag toward the D3 location. You can either download the library from Github or point to its hosting source, which is <script src="//d3js.org/d3.v3.min.js" charset="utf-8"></script>. This will load D3 and you can start coding in your Javascript file using D3's syntax. More on that later.

Selections

The preceding code featured some lines of code with the word "select" in them. This is one of the types of selectors in D3. Generally, selectors select a page element and tell D3 this is where you are going to place data.

More specifically, D3 handles groups of related elements called selection. Working with these groups of elements en masse gives your selection power and allows you to manipulate a single element or many at once without restructuring your code or having to run loops and other control flow code. In short, it’s a cleaner way to work with data.

According to D3js.org: “A selection is an array of elements pulled from the current document.”

There are two methods of selecting elements in D3. They are "select" and "selectAll"

These methods allow you to accept selector strings. The first method selects only the first matching element, while the latter selects all matching elements in document traversal order. These methods can allow you to import third-party libraries with nodes.

Let's go through how these selectors can be used.

d3.select(selector)
This line takes the first element that matches the specified selector string and returns a single-element selection. If nothing matches in the current document the selection returns empty. If multiple elements match the selector, only the first matching element (in document traversal order) will be selected.

d3.selectAll(selector)
This takes in all elements that match the specified selector. The elements will be selected from top-to-bottom in the document. If no nothing matches this line of code, the returns comes back empty.

d3.select(node)
This selects a specified node. This comes in handy when you already have referenced to a node, such as d3.select(this) within an event listener or a global event such as document.body.

d3.selectAll(nodes)
This selects the specified array of elements. If you’ve already referenced to nodes, such as d3.selectAll(this.childNodes) within an event listener, or a global such as document.links, this can be handy. The nodes argument doesn't have to be an array. Any pseudo-array that can be forced into an array will work.

Remember: Selections are arrays of elements.

This is where things begins to make sense in terms of D3 marrying four different languages together.

Here are just some of the HTML, CSS or SVG elements you can select.

  • Tag ("div")
  • Class (".awesome")
  • Unique identifier, known as id, ("#foo")
  • Attribute ("[color=red]")
  • Containment ("parent child")
  • G element, which is a container used to group other SVG elements you draw

Selectors can also be intersected (".this.that" for logical AND) or unioned (".this, .that" for logical OR).

Loops or recursive functions aren’t needed because D3 allows you to modify entire arrays at once. This eliminates the need to looping individual elements over and over again. You can still loop if you wish to. D3 has an each operator that invokes an arbitrary function so elements can be accessed directly. This is of particular importance because of the data visualization I've created later in this wiki.

We'll take a look at how selectors work in action later.

The nuance of D3 comes from the fact that selections are grouped. Instead of arrays being one-dimensional, each selection is an array of array of elements.

The advantage of this is that it preserves the hierarchical structure of subselections. This can usually be overlooked. However, in single-element selection this looks like [[node]] instead of just node.

Operators

Operators tell data to behave in a specific manner. You first select an element and then tell it do stuff with an operator. They allow you to display data. Selectors allow you to select, group and categorize that data.

Operators can get or set attributes, styles, properties, HTML and text content. Attribute values are specified as constants or functions. Functions will be evaluated for each element.

They must return the current selection, so you can chain multiple operators together in a concise statement. In other words, your operators must make sense in order for the computer to run them into multiple chains that simplify the overall output, your visualization or a component of that visualization.

The D3 documentation lists all of the different operators.

Appends

Appends insert a new element in a selection.

For example:

var g = svg.selectAll("g")
.data(function (d) { return pie(d.percent); })
.enter().append("g");

g.append("text")
.attr("transform", function(d) { return "translate(" + arc.centroid(d) + ")"; })
.attr("dy", ".35em")
.attr("class", "arcLabel")
.style("text-anchor", "middle")
.text(function(d) { return d.data.percent == 0 ? "" : d.data.percent.toFixed(1) + "%" ; });

Data joins

Say you want some circle elements to correspond to data where each circle is one datapoint. You want one circle per datum. Instead of instructing D3 to create circles, tell D3 that the selection "circle" should correspond to data. This concept is called the data join.

Joins let you target operations to specific states. For example, you can set constant attributes (such as the circle’s radius, defined by the "r" attribute) on enter rather than update. By reselecting elements and minimizing DOM changes, you vastly improve rendering performance! These are all specific states the data takes on.

Enters, updates, exits

Selections can be joined to data, which can be made available to operators for visualization. Enters, updates and exits represent data states. These three

By joining data, enter and exit subselections are created so that you can easily add or remove elements in response to changes in data.

For more information about D3, its syntax and functions, go to D3js.org

Limitations of D3

D3 has a few a limitations. DOM manipulation can be slow for a large number of data entries. What this means is that the more data you have that you are trying display on your page at one time, the slower it will load. DOM stands for Document Object Model, which is the programming interface for HTML, CSS, XML and SVG.

SVG's performance will lag when it has to display a large volume of elements at once.

Another hinderance in D3 is how opaque some of the functions and syntax can be because of how it is a modified version of JavaScript. If you have a strong foundation in JS, you should be fine.

I personally ran into issues not knowing everything about JavaScript. I spent a lot of time reading the D3 documentation and browsing the Stacked Overflow forums. You should know how to use HTML and CSS. Knowing SVG is not necessary because there are plenty of D3 examples that use it where you can figure out what's going on. Creating my visualization was a lot of trial and error, to be honest.

Walkthrough

What you need

In order to create this visualization, you need to collect the data first.

I pulled my data from mivavolleyball.com. I pulled this data set from the standings page. I scraped it off the HTML with Google Sheets. I dropped the URL into a cell on the sheet and then ran the =ImportHTML function and told it to pull all data located within the "Table" tags of the page's HTML.

The No. 1 in the function indicates index on the HTML. It's telling the sheet to pull data from Table No. 1.

I had an issue where the Win/Loss Record was being imported as a date. To fix this, I ran the =Split function and assigned a delimeter of "/" in order to tell it to split the date into 3 separate columns based on "/". Now, I have the Win/Loss record displayed cleanly. Then, I divided the wins from the total number of games and created columns for Win % and Loss %. I exported this information into a CSV file because D3 can pull external data from a CSV or TSV file that's in the local directory. I uploaded this csv file on my server and moved on to the visualization part. I only exported the conference numbers for simplicity.

Setting up your HTML

Declare your document type. Establish your header and body tags. You'll want to create div with a custom class name inside your body. The reason for this is that you need to designate a space where your D3 charts will generate once your visualization goes live. If you just drop the scripts loose inside the body tags, the charts will generate at the body of the page.

Inside your divs, you'll need to create two script tags. One imports the D3 library, <script src="//d3js.org/d3.v3.min.js" charset="utf-8"></script>, and the other points toward your external JS file. See the below image for an example.

The HTML for our data viz is underneath the HTML comment. The text above is the chart title and chatter. The em tags beneath the div are for linking to sources. The next thing we'll look at is the full JavaScript that we've linked to in the div. We'll get to the CSS file after that.

Not a whole lot of JavaScript, right? That's what's great about D3 is that minimizes the amount of code you have write for the nine charts you just saw. This donut chart is about 90 lines of JavaScript in total. There are two many groupings of the Javascript here. The first part is a number of variable declarations and the second part is the creation of our charts, which starts with pulling data from the selection d3.csv. The way these two groupings interact is that the d3.csv part runs a series of functions and commands that rely on the individually defined variables we have up top. The first group is essentially a loose collection of variables and the second part is unified system of selections. Try to think of the first group as the United States and the second group as a country like Germany. One is a loose collection independent governments and the other is centralized government with some provinces.

Let's delve into the first group and then we'll go back and review the CSS relevant to the donut charts.

As you go line-by-line, you'll realize we are defining the dimensions of our donut charts. At first glance, it looks like it's for a singular chart. It's not. We're establishing general parameters that can plugged into reiterated functions

Radius and padding determine the entire size of our donut chart and the padding surrounding it. Color determines how many colors we'll be available for our differnet data values/columns. We're selecting the colors for each of the slices on the donut. Arc is establishing the measurements needed to create a donut chart. It states that outerRadius is our previously define radius and that innerRadius is radius minus 35. Pie declaring a variable that tells D3 we want to create a pie chart that will become a donut chart because of the inner and outer Radii.

These are the variables that will be used over and over again to create a unified set of donut charts.

They are items that will appear in our D3 code. That's what's great and confusing about D3. You can define CSS classes inside JavaScript. The only CSS class here that's in our HTML is .donut because that's our div containing the JavaScript.

We've gone over the variables that will define how the series of charts will be created and also looked at the CSS that will stylize them. Let's look at how we pull this data in and tell D3 what we want. We'll start with a survey of the entire code that actually create the charts and legend.

Examples of use in journalism

D3 is a tool widely used in journalism. I felt The New York Times has some of the best examples of its uses in the industry.

Paths to the White House - The New York Times

This tool allows the user to explore the possibilities and alternate outcomes of the 2012 presidential election. The preface to this tool is that President Obama won a clear victory, but his popular vote margin in several swing states was thin. The premise here is to examine what Obama or Romney's path to the White House would have looked like if certain states had voted differently than they actually did.

The dataset in this tool includes the percent each candidate won or lost a swing state. This data is used as a label up top. The core visualization is of three columns—Swing State, did Obama win and did Romney win. This creates a series of yes or no values that can be used to calculate all possible pairings. The result is the title "512 Paths to the White House." The tool displays all of these paths by default. The user can alter the visualization by selecting a winner of a state at the top. This will narrow the paths to the Oval Office. It shows how numerous Obama's paths were versus Romney's.

D3 was a good choice for this dataset because of its ability to command data. Remember: It's about telling D3 what you want versus how to do it.

Drought Extends, Crops Wither - The New York Times

This tool by the New York Times focuses on showing how 2012's drought rivaled the 1988 drought and how droughts affected the economy. This looks at the varying effects of the summer heat and drought on crops critical to the farm economy.

This tool visualizes crop condition, price received and yield for a specific crop and crop size and location for a number of key U.S. crops from corn to cotton. The tool does a forEach command like in my visualization to create separate charts for each crop. The tool allows the reader to hover over a specific date and see how the nation was affect by heat and drought both in terms of crop yield and economic production. The tool also points at key moments and outliars in each chart for each crop. It offers context as to why certain things happened.

This data was a good fit for D3 because of D3's ability to group similar elements, separate them when necessary and iterate certain functions over and over again with minimal overhead. This is partially done with the forEach function. Everything on this page is unified in terms of design and data. What separates the different charts is the type of crop. D3 generalizes elements and data in order to make manipulating data easier. This is a prime example of that concept.

How the Chicago Public School District Compares - The New York Times

This tool is a little more complex than the previous ones. It's not minimal it's design or layout. There's a lot of information in this data and in the context of this data. There's a lot going on here. However, this is a good example of how to manipulate large data sets and make it digestible. While it's not simple, it is well-designed and context is effectively provided.

This tool focuses on breaking down the issues that are important to the Chicago public schools and attempts to provide context as to why the Chicago Teachers Union might have gone on strike. It looks at a variety of data sets such as annual salaries, how many hours teachers work and if test scored are used in evaluations. The tool allows you to explore how Chicago pairs up to other major cities across the country in terms of these data sets. Are Chicago schools teachers paid the most? You can figure that out. The tool comes with chatter explaining what the various interactive columns mean.

This data set was a good fit for D3 because of how large and dimensional it was. D3 works well with generalizing sets of data and allowing the user to manipulate it through a front end. This tool exemplifies this.
